// the problem

vllm serve <model> leaves
3-to-8× tokens/sec and 5-to-15× tail latency on the table.

The same model on the same GPU can differ by 3-8× in throughput and 5-15× in P99 latency depending on how its ~40 knobs are set. Every default is a guess at the wrong workload, the wrong hardware, the wrong quantization, the wrong attention backend — and your GPUs pay the bill.

// the search space

Six orthogonal axes. A mega-kernel configuration.

We model vLLM as a compilation target, not a black box. Each axis is reasoned about independently — with hard compatibility constraints — and then composed.

01 Parallelism ~8 knobs

TP × PP × DP × EP. NVSwitch favors TP. PCIe favors PP. MoEs (DeepSeek-V3.2, Qwen3.5-A17B) converge on --dp N --enable-expert-parallel.

tensor_parallel_size pipeline_parallel_size data_parallel_size enable_expert_parallel
02 Quantization ~12 knobs

Weights × activations × KV cache, separately. NVFP4 on Blackwell often beats FP8. --kv-cache-dtype fp8_e4m3 nearly doubles effective KV capacity.

FP8 W8A8 NVFP4 AWQ-INT4 GPTQ-INT4 Marlin Machete DeepGEMM
03 Attention backend ~5 backends

FlashAttention 3 dominates on Hopper for long seq. FlashInfer wins on variable-length paged decode. Gemma-4 / GPT-OSS force backend by capability, not speed.

FlashAttention 3 FlashInfer Triton-native xFormers cuDNN + TMA
04 Speculative decoding ~10 methods

Single highest-leverage knob for decode-bound workloads. EAGLE-3, P-EAGLE, MTP heads (deepseek_mtp, qwen3_next_mtp, glm4_moe_mtp).

EAGLE-3 P-EAGLE Medusa n-gram suffix decoding MLP speculator MTP × 7 variants
05 Scheduler & memory ~9 knobs

Sharp non-linearities. Chunked prefill chunk-size too small kills prefill; too large starves decode. CUDA graph mode interacts with speculation availability.

max_num_seqs max_num_batched_tokens gpu_memory_utilization chunked_prefill cudagraph_mode prefix_caching
06 Compiler & runtime ~8 knobs

torch.compile passes, fusion, NCCL tuning. VLLM_USE_V1, NCCL_IB_HCA, buffer sizes — can shift P99 latency by 20% with no other changes.

torch.compile VLLM_USE_V1 VLLM_ATTENTION_BACKEND NCCL_IB_HCA NCCL_P2P_LEVEL
Composed search dimension: ~184,320 raw configs · pruned to ~50 viable in <1s

// the search

Hierarchical, multi-fidelity, LLM-guided.

Four phases. Cheapest first. Each phase prunes the candidate set before the next phase spends a single extra GPU-second.

01
Feasibility pruning
cost: 0 GPU-sec

Deterministic. Compatibility graph + HBM accounting. Typically prunes the space by 1-2 orders of magnitude.

02
Anchor selection
cost: 1 LLM call

Planner produces 3-5 anchors: the recipe baseline, +speculation, +DP/EP, +exploratory. Each anchor carries an explicit hypothesis.

03
Multi-fidelity search
cost: ~1-4 GPU-hr

Smoke (60 s) → short (5 min) → BoTorch local search. LLM vetoes proposals that violate recipe priors.

04
Long-run validation
cost: ~2-6 GPU-hr

30-min steady state, spike, soak, chaos, reproducibility re-run. Only certified configs ship.

Search budget
fast mode (warm-start, KB hit) < 1 hr
typical full tune 6-12 hr
research mode (new GPU / model / vLLM major) 24+ hr · feeds KB

Tell us the use case and we apply the right priors. Or hand us your production traces and we fit the distribution.

Agentic coding

Cursor / SWE-agent / Claude Code style

  • priorsaggressive prefix caching, large blocks, EAGLE-3, moderate concurrency
  • expected gain19.4% lower $/Mtok

Reasoning model serving

o1-style · R1-style · GPT-OSS reasoning
  • prioritydecode tok/s · P99 time-to-full-response
  • priorsFP8 KV mandatory, MTP=1 / EAGLE-3, extreme batch sizes
  • expected gain2-3× decode throughput

Interactive chat

consumer · enterprise assistants
  • priorityTTFT P99 < 500 ms · ITL P99 < 100 ms · goodput
  • priorsbalanced batch, prefix cache, FP8 KV, P-EAGLE, chunked prefill
  • expected gain2× goodput at SLO

Long-context RAG

document QA · agentic search
  • priorityprefix cache hit rate · prefill throughput
  • priorslarge block size, aggressive FP8 KV, tuned chunked prefill, DP+EP for MoE
  • expected gain10×+ TTFT on cache-hit

Bulk synthetic / labeling

offline batch · data generation
  • priorityraw output tok/s · TTFT irrelevant
  • priorsmax_num_seqs > 200, batched_tokens 16K-32K, NVFP4, DP+EP
  • expected gain3-4× throughput

Real-time voice

streaming · agentic phone
  • priorityTTFT P99 < 200 ms · ITL jitter under 30 ms
  • priorssmall batches, latency-optimized attention, async sched, disable chunked prefill if jitter
  • expected gainconsistent sub-200ms TTFT
// validation

Three concentric rings. No certification without all three.

Most auto-tuners stop at performance. We measure the things that silently break in production — quality drift from speculation, NCCL deadlocks in soak, KV fragmentation under long-tail traffic.

RING 1

Performance

TTFT/ITL/TPOT P50/P95/P99, goodput at SLO, prefix-cache hit rate, KV occupancy, per-position speculative acceptance. Bootstrap CIs — not point estimates.

TTFT P99 < 500 ms
ITL P99 < 100 ms
Steady-state detect ≥ 70% plateau
CI confidence bootstrap 95%
RING 2

Correctness

The ring most tuners skip. INT4 / NVFP4 / speculative decoding can silently degrade output quality. We probe with a fixed regression suite against a BF16 reference.

MMLU-Pro
GSM8K + HumanEval
MT-Bench + IFEval
Long-context needle-in-haystack
Tool-calling correctness
Quality regression flag at ≥ 1% composite
RING 3

Operational chaos

A config that wins in lab conditions can fall over in production. We run the chaos suite, then run the entire tune again from scratch to prove it wasn't a fluke.

2-hour load soak (mem leak, NCCL deadlock)
Spike test (5× rate for 60s)
Long-tail (10× normal seq len)
Cold-start time-to-first-success
Reproducibility re-run from fresh server

What you get: a Certified Recipe

Not a benchmark result. A reproducible, explained, drop-in artifact — in the exact recipes.vllm.ai schema, with Docker, Helm, and a tuning report written by the Planner.

  • recipe.yaml
    vLLM recipes schema · drops into recipes.vllm.ai ecosystem
  • docker-compose.yml
    env vars, volumes, resource requests, liveness probes pre-filled
  • helm-chart/
    production-stack template, K8s-ready, multi-tenant
  • tuning_report.md
    written by Planner · assumptions · trade-offs · fragility points
  • provenance.json
    vLLM commit · torch · CUDA · NCCL · driver · all benchmark traces

// positioning

TokenPeak is a wrench. autoinference is a maintenance system.

TokenPeak proved that auto-tuning vLLM's scheduler beats hand-tuned defaults. We agree. But the actual search space is 10-20× larger, and the deliverable should be a recipe an ops team can ship, not a benchmark CSV.
